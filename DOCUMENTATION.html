1) Feature Inventory (implemented)

ICS (Operational Dashboard + POS + Admin)

  • Authentication: email/password + restaurant code login; password reset page.
  • RBAC: protected routes based on permissions.
  • Dashboard: live stats (active orders, tables occupied, kitchen load, sales summary).
  • POS: categories, menu items, cart, discount/tax, place order (writes orders + orderItems + table status).
  • Tables: table list, open order info per table, “Open POS” shortcut.
  • Menu Management: CRUD categories/menu items; image upload to Storage.
  • Staff Management: add/remove staff, drafts, document uploads, CSV export, audit logs.
  • Restaurant Logo: editable logo upload with validation + crop/resize; stored in Storage and referenced from restaurant doc.

KDS (Kitchen Display System)

  • Authentication: shared login flow.
  • Kitchen queue: real-time order items by station, sound notification for new orders.
  • Status updates: PREPARING/READY updates orderItems; best-effort parent order kitchenStatus update.

Platform / Infrastructure

  • Firebase Hosting: multi-site targets (ics + kds) via firebase.json.
  • Firestore rules + Storage rules enforce restaurant scoping + RBAC.
  • Callable Cloud Function: uploadRestaurantLogo (secure logo upload).
  • Google Cloud credential smoke-test script for service-account validation.

3) Architecture

  • Auth model: Firebase Auth (email/password) + restaurant code lookup (Firestore).
  • Tenant scoping: all business docs contain restaurantId.
  • RBAC: permissions are derived from roles (roles collection). The user document denormalizes permissions for rules.
  • Real-time: pages use Firestore onSnapshot subscriptions for live updates.
  • UI: React + Tailwind utility classes; shared UI kit in shared/components.

4) Data Model & Collections

Primary types are defined in: shared/firebase/types.ts

Core collections

  • users – user profile (document id = Firebase Auth UID); includes restaurantId, roleIds, permissions (denormalized).
  • restaurants – tenant metadata; includes code used for login and optional logoStoragePath.
  • roles – role name + permissions, scoped by restaurantId.
  • orders – OPEN/CLOSED; includes kitchenStatus; totals.
  • orderItems – items belonging to orders; kitchenStatus per item.
  • tables – dine-in table state.
  • categories, menuItems – menu management.
  • staff – staff records (admin only), with subcollection staff/{id}/documents.
  • auditLogs – append-only audit entries for staff/doc events (admin only).

5) Security (Rules + RBAC)

  • Firestore rules: firestore.rules
  • Storage rules: storage.rules

RBAC flow

  • Login retrieves roles for the user and denormalizes permissions onto users/{uid}.
  • UI gates routes via ProtectedRoute and usePermissions().
  • Rules enforce permissions for reads/writes.

10) Troubleshooting (common issues)

FirebaseError: auth/invalid-api-key

  • Cause: API key referrer restrictions missing localhost/prod domains.
  • Fix: add http://localhost:5173/*, http://localhost:4173/*, http://localhost:4174/* to allowed referrers.

“Restaurant code not found”

  • Ensure restaurants/{id} has field code matching the login input.

“Missing or insufficient permissions”

  • Ensure users/{uid}.restaurantId matches and role contains required permission key.
